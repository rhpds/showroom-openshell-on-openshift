= Level 4: Policy Profiles
:navtitle: Level 4: Policy Profiles
:toc:
:toc-placement: preamble
:icons: font

This level runs the same GitHub inspection prompt through three OpenClaw instances — each with a different policy posture — and compares the results side by side.
Budget about 35 minutes, not counting initial image pulls.

== Learning Objectives

* Compare deny-all, managed, and endpoint-allowlist policies against an identical prompt and explain the observed differences
* Distinguish between endpoint-only reachability and REST read-only enforcement at the policy level
* Use a shared RUN_ID to tie comparison results to the correct session and sandbox
* Read effective policy from a running Sandbox resource and match it to observed tool output

== Starting State

This level assumes Level 3 is complete with all verification checks passing.
The following resources must exist before you begin:

[source,bash,role="execute",subs="attributes"]
----
kubectl -n {openshell_namespace} get statefulset openshell-gateway
kubectl -n {openclaw_namespace} get secret my-llm-key
----

Both commands must return resources.
If either fails, return to the level where it was created before continuing here.

=== Step 3: Create the managed profile

The managed profile allows GitHub API calls only from named binaries using `GET` or `HEAD` methods.
This enforces read-only access at both the HTTP method and calling-binary level.

[source,bash,role="execute"]
----
cat > /tmp/policy-profiles/overlays/managed/policy.yaml << 'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: openclaw-managed-policy
  namespace: openclaw
data:
  policy.yaml: |
    version: "1"
    name: github-api-read-only
    description: REST read-only access to GitHub API, restricted to named binaries
    network:
      rules:
        - name: github_api_read_only
          endpoints:
            - host: api.github.com
              port: 443
          methods:
            - GET
            - HEAD
          binaries:
            - curl
            - wget
EOF
----

== Exercise 4: Run the Controlled Prompt

This exercise sends the same prompt to each profile instance in sequence.
The prompt instructs the agent to run exactly one shell command and stop — no additional tool calls, no substitution from the model's own knowledge.

WARNING: Do not modify the prompt text between profiles. Any change invalidates the comparison.

. Set the fixed prompt and generate a run ID:
+
[source,bash,role="execute"]
----
FIXED_PROMPT="$(cat <<'EOF'
Use the shell execution tool exactly once to run this exact command, then stop:
curl -sS -o /sandbox/level4-github.json -w '%{http_code}' \
  https://api.github.com/repos/NVIDIA/OpenShell \
  > /sandbox/level4-http-code || true
Do not substitute your own knowledge for tool output.
EOF
)"

RUN_ID="$(date +%s)-$(openssl rand -hex 3)"
printf 'Comparison run: %s\n' "${RUN_ID}"
----
+
Record the `RUN_ID` value printed to the terminal.
Every subsequent command in this exercise uses it.

. Run the prompt through each profile:
+
[source,bash,role="execute"]
----
for profile in deny managed demo-allowlist; do
  case "${profile}" in
    deny) session_code=d ;;
    managed) session_code=m ;;
    demo-allowlist) session_code=a ;;
  esac
  session_key="agent:main:l4-${session_code}-${RUN_ID}"
  pod="$(kubectl -n openclaw get pod \
    -l "app.kubernetes.io/name=openclaw-${profile}" \
    --field-selector=status.phase=Running \
    -o jsonpath='{.items[0].metadata.name}')"
  test -n "${pod}"
  printf '\n--- Running profile: %s (pod: %s) ---\n' "${profile}" "${pod}"
  kubectl -n openclaw exec "${pod}" -- \
    node /app/openclaw.mjs agent --local \
      --session-key "${session_key}" \
      --message "${FIXED_PROMPT}" --json | \
    tee "/tmp/level4-${profile}-${RUN_ID}.json"
done
----
+
Each run takes 20 to 60 seconds depending on model latency.
The loop prints the session key and pod name before each run so you can track progress.

=== Verify

Confirm output files were created for all three profiles:

[source,bash,role="execute"]
----
ls -lh "/tmp/level4-deny-${RUN_ID}.json" \
        "/tmp/level4-managed-${RUN_ID}.json" \
        "/tmp/level4-demo-allowlist-${RUN_ID}.json"
----

Each file must exist and have a non-zero size.
An empty or missing file means the agent run failed before producing output.

== Exercise 5: Correlate Descendant Sandboxes

Each agent run created one Sandbox custom resource in the `openshell` namespace.
This exercise maps each profile name to its Sandbox name using the RUN_ID timestamp, so the artifact reads in Exercise 6 target the correct sandbox.

. Collect the Sandbox name for each profile and write a mapping file:
+
[source,bash,role="execute"]
----
(
RUN_STAMP="${RUN_ID%%-*}"
: >"/tmp/level4-sandboxes-${RUN_ID}"
for profile in deny managed demo-allowlist; do
  case "${profile}" in
    deny) session_code=d ;;
    managed) session_code=m ;;
    demo-allowlist) session_code=a ;;
  esac
  matches="$(kubectl -n openshell get \
    sandboxes.agents.x-k8s.io \
    -o jsonpath='{range .items[*]}{.metadata.name}{"\n"}{end}' | \
    grep "^openclaw-agent-main-l4-${session_code}-${RUN_STAMP}-" || true)"
  count="$(printf '%s\n' "${matches}" | awk 'NF {n++} END {print n+0}')"
  if test "${count}" != 1; then
    printf 'Expected one Sandbox for %s; found %s\n' \
      "${profile}" "${count}" >&2
    exit 1
  fi
  printf '%s %s\n' "${profile}" "${matches}" | \
    tee -a "/tmp/level4-sandboxes-${RUN_ID}"
done
)
----

. Review the profile-to-sandbox mapping:
+
[source,bash,role="execute"]
----
cat "/tmp/level4-sandboxes-${RUN_ID}"
----

=== Verify

[source,bash,role="execute"]
----
wc -l < "/tmp/level4-sandboxes-${RUN_ID}"
----

The output must be `3`.

If you see a count other than `3`, check that the `RUN_ID` variable is still set in your shell and that all three agent runs in Exercise 4 completed without error.

== Exercise 6: Judge Tool Output

Read the HTTP status code and repository name that each agent wrote into its sandbox filesystem.
These values come directly from the `curl` command output — not from the model's text response.

. Read the artifacts from each sandbox:
+
[source,bash,role="execute"]
----
for profile in deny managed demo-allowlist; do
  sandbox="$(awk -v profile="${profile}" \
    '$1 == profile {print $2}' "/tmp/level4-sandboxes-${RUN_ID}")"
  test -n "${sandbox}"
  pod="$(kubectl -n openclaw get pod \
    -l "app.kubernetes.io/name=openclaw-${profile}" \
    --field-selector=status.phase=Running \
    -o jsonpath='{.items[0].metadata.name}')"
  printf '\n=== %s (sandbox: %s) ===\n' "${profile}" "${sandbox}"
  kubectl -n openclaw exec "${pod}" -- \
    openshell sandbox exec --name "${sandbox}" \
    --gateway-endpoint \
    "http://openshell.openshell.svc.cluster.local:8080" \
    -- sh -lc 'printf "status="; cat /sandbox/level4-http-code; \
      printf " repository="; \
      if test -f /sandbox/level4-github.json; then \
        sed -n '\''s/.*"full_name"[[:space:]]*:[[:space:]]*"\([^" ]*\)".*/\1/p'\'' \
        /sandbox/level4-github.json | head -n 1; fi; printf "\n"'
done
----

== Exercise 7: Inspect Effective Policy and Denial Evidence

The final exercise reads the effective policy from each Sandbox resource and checks the gateway logs for the denial record that the deny profile generated.

. Read the effective policy from each sandbox:
+
[source,bash,role="execute"]
----
for profile in deny managed demo-allowlist; do
  sandbox="$(awk -v profile="${profile}" \
    '$1 == profile {print $2}' "/tmp/level4-sandboxes-${RUN_ID}")"
  test -n "${sandbox}"
  printf '\n=== Effective policy: %s ===\n' "${profile}"
  kubectl -n openshell get sandbox "${sandbox}" \
    -o jsonpath='{.status.effectivePolicy}' | \
    python3 -m json.tool 2>/dev/null || \
  kubectl -n openshell get sandbox "${sandbox}" \
    -o jsonpath='{.status.effectivePolicy}'
  printf '\n'
done
----

. Check the OpenShell gateway logs for denial evidence from the deny profile run:
+
[source,bash,role="execute"]
----
kubectl -n openshell logs statefulset/openshell-gateway \
  --since=15m | grep -i "deny\|api\.github\.com" | tail -20
----

Expected: at least one log line naming `api.github.com` with a denial reason such as `network_policy_deny` or `no_matching_rule`, produced during the deny-profile agent run.
